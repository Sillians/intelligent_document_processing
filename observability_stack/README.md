# Observability Stack

## Core Purpose
Monitor the Intelligent Document Processing platform with Prometheus metrics, Grafana dashboards, and Prometheus alert rules.

The stack currently focuses on reliable service-level monitoring for the completed IDP pipeline:

```text
ingestion-service
preprocess-worker
ocr-service
layout-service
classifier-router-service
extraction-service
validation-service
human-review-console
delivery-service
evaluation-service
```

## Implemented Core Functionalities
- Prometheus scrape configuration for all FastAPI-based IDP services.

- Shared `/metrics` endpoint scraping through `prometheus_client` instrumentation.

- Service availability monitoring via Prometheus `up` targets.

- HTTP request rate, status code, and latency metrics.

- Prometheus alert rules for service-down, missing scrapes, HTTP errors, p95 latency, Temporal queue backlog, stale queue telemetry, and low disk space.

- Grafana datasource provisioning for Prometheus.

- Grafana dashboard provisioning for the `IDP Overview` dashboard.

- Docker healthchecks for Prometheus and Grafana.

- Operational links from Grafana to Prometheus, MLflow, Temporal UI, and Label Studio.

- Node Exporter metrics for staging disk-capacity alerts.

- Deployment-time observability verification with evidence artifacts.

## Metrics Emitted By Services
All instrumented FastAPI services expose:

```text
GET /metrics
```

Shared metrics:

```text
http_requests_total{service,method,path,status}
http_request_duration_seconds_bucket{service,method,path,le}
http_request_duration_seconds_count{service,method,path}

idp_temporal_task_queue_backlog{namespace,task_queue,task_type}
idp_temporal_task_queue_pollers{namespace,task_queue,task_type}
node_filesystem_avail_bytes

```

These are generated by [metrics.py](../intelligent_document_processing/shared/idp_common/metrics.py).


## Prometheus Configuration
Primary files:

```text
infra/prometheus/prometheus.yml
infra/prometheus/alerts.yml
```

Prometheus scrape groups:
- `prometheus`: Prometheus self-monitoring.
- `idp-api-services`: all FastAPI IDP services exposing `/metrics`.
- `idp-workflow-queue`: Temporal queue metrics from the workflow orchestrator.
- `node-exporter`: filesystem and host/VM capacity metrics.

Validate loaded targets in the UI:

```text
http://localhost:9090/targets
```

Useful PromQL checks:

```promql
up{tier="idp-pipeline"}
sum by (service) (rate(http_requests_total[5m]))
sum by (service) (rate(http_requests_total{status=~"5.."}[5m]))
histogram_quantile(0.95, sum by (service, le) (rate(http_request_duration_seconds_bucket[5m])))
sum by (namespace, task_queue) (idp_temporal_task_queue_backlog)
node_filesystem_avail_bytes{mountpoint="/"} / node_filesystem_size_bytes{mountpoint="/"}
```

## Alert Rules
Alert rules are defined in:

```text
infra/prometheus/alerts.yml
```

Implemented alerts:
- `IDPServiceDown`: a pipeline service cannot be scraped.
- `IDPMetricsScrapeMissing`: Prometheus has no pipeline scrape targets.
- `IDPHigh5xxRate`: a service is returning sustained server errors.
- `IDPAnyCritical5xxRate`: critical path services have sustained server errors.
- `IDPHighClientErrorRate`: a service is receiving many invalid or unauthorized requests.
- `IDPHighP95Latency`: service p95 latency is above 5 seconds.
- `IDPNoRecentTraffic`: a service has no non-health traffic for 30 minutes.
- `IDPTemporalQueueBacklogHigh`: combined workflow/activity backlog exceeds 10 tasks.
- `IDPTemporalQueueMetricsStale`: queue collection has not succeeded recently.
- `IDPTemporalQueueMetricsMissing`: no Temporal backlog series exist.
- `IDPNodeExporterDown`: disk telemetry cannot be scraped.
- `IDPDiskSpaceLow`: root filesystem free space is below 15%.
- `IDPDiskSpaceCritical`: root filesystem free space is below 5%.

Validate alert expressions and firing behavior with the pinned Prometheus image:

```bash
docker run --rm \
  -v "$PWD/infra/prometheus:/etc/prometheus:ro" \
  --entrypoint promtool \
  prom/prometheus:v2.55.1 \
  test rules /etc/prometheus/alerts.test.yml
```

## Staging Verification

Staging binds the three observability UIs to loopback. Configure:

```text
STAGING_PROMETHEUS_URL=http://127.0.0.1:9090
```

The deploy workflow starts and health-checks the stack automatically. Run the
same verification manually:

```bash
python3 scripts/staging_operational_drill.py observability \
  --prometheus-url http://127.0.0.1:9090 \
  --alertmanager-url http://127.0.0.1:9093 \
  --grafana-url http://127.0.0.1:3000
```

The command requires all ten pipeline targets, worker queue telemetry, root
filesystem metrics, Grafana health, Alertmanager status, and every required
alert rule. Evidence is written under `artifacts/staging/observability/`.

Inspect active alerts:

```text
http://localhost:9090/alerts
```

Inspect Alertmanager routing and notification state:

```text
http://localhost:9093
```

Local development note: `IDPNoRecentTraffic` can fire when the pipeline is idle. Keep it at `severity=info` or disable it for quiet local environments.

## Email Alert Delivery
Prometheus sends firing alerts to Alertmanager, and Alertmanager sends grouped email notifications.

Configuration files:

```text
infra/alertmanager/alertmanager.yml.template
infra/alertmanager/render-and-run.sh
infra/alertmanager/templates/idp-email.tmpl
```

Configured recipient:

```env
ALERT_EMAIL_TO=basil.sillians@gmail.com
```

SMTP configuration:

```env
ALERT_EMAIL_FROM=basil.sillians@gmail.com
ALERT_EMAIL_SMARTHOST=smtp.gmail.com:587
ALERT_EMAIL_USERNAME=basil.sillians@gmail.com
ALERT_EMAIL_PASSWORD=<gmail-app-password-or-smtp-password>
ALERT_EMAIL_REQUIRE_TLS=true
ALERT_EMAIL_REPEAT_INTERVAL=4h
```

For Gmail, `ALERT_EMAIL_PASSWORD` should be a Gmail App Password, not the normal account password. Create it from Google Account security settings after enabling 2-step verification.

Apply email configuration:

```bash
docker compose up -d --force-recreate alertmanager prometheus
```

Validate Alertmanager configuration:

```bash
docker compose exec alertmanager amtool check-config /tmp/alertmanager.yml
```

Send a safe test alert directly to Alertmanager:

```bash
curl -X POST http://localhost:9093/api/v2/alerts \
  -H "Content-Type: application/json" \
  -d '[{
    "labels": {
      "alertname": "IDPEmailSmokeTest",
      "severity": "info",
      "category": "test",
      "service": "observability-stack"
    },
    "annotations": {
      "summary": "IDP email alert smoke test",
      "description": "This is a manual test alert generated to verify Alertmanager email delivery."
    },
    "startsAt": "'$(date -u +"%Y-%m-%dT%H:%M:%SZ")'"
  }]'
```

If no email arrives:
- Check `docker compose logs --tail=200 alertmanager`.
- Confirm `ALERT_EMAIL_PASSWORD` is set and valid.
- Confirm Gmail App Passwords are enabled.
- Confirm the alert is visible at `http://localhost:9093`.
- Check spam/promotions folders.

## Troubleshooting

Check Prometheus config syntax from the running container:

```bash
docker compose exec prometheus promtool check config /etc/prometheus/prometheus.yml
```

Reload Prometheus after config changes:

```bash
curl -X POST http://localhost:9090/-/reload
```

If a service target is down:

```bash
docker compose ps <service-name>
docker compose logs --tail=200 <service-name>
curl http://localhost:<published-port>/health
```

If Grafana does not show the dashboard:

```bash
docker compose logs --tail=200 grafana
docker compose restart grafana
```

If Prometheus does not scrape a service:
- Confirm the service exposes `/metrics`.
- Confirm the service is on the `idp` Docker network.
- Confirm Prometheus targets use the container DNS name and internal port.
- Check `http://localhost:9090/targets` for the scrape error.

If Prometheus alerts are not reaching Alertmanager:

```bash
docker compose exec prometheus wget -qO- http://localhost:9090/api/v1/alertmanagers
docker compose logs --tail=200 prometheus
docker compose logs --tail=200 alertmanager
```

## Security and Compliance
- Grafana signup is disabled by default.
- Replace default Grafana credentials before non-local use.
- Do not expose Prometheus or Grafana directly on public networks without authentication and TLS.
- Avoid adding high-cardinality labels such as document IDs, tenant IDs, filenames, or user IDs to metrics.
- Application logs and future traces should avoid raw OCR text and extracted PII.

## Non-Goals
- Distributed tracing.
- Centralized log indexing.
- Business-level quality dashboards from MLflow metrics.

These can be added after the core Prometheus and Grafana baseline is stable.
